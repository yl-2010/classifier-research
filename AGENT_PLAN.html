Student Notes Subject Classifier

Build a useful student note-taking website that auto-files pasted notes into subjects, while running a real research comparison: prompted LLM vs base BERT vs fine-tuned BERT, with an orchestrator LLM that adjudicates and can invent Other / custom subjects. This plan captures the full design discussion that evolved from the UK headlines DistilBERT domain-adaptation project.

0. Lineage — what this builds on

Prior sibling project (uk_headlines_practical, separate repo): DistilBERT topic classification with domain shift (AG News → UK BBC), Naive Bayes baselines, embedding similarity, novel-words analysis, and a GitHub Pages results site. Key lessons to carry forward:

  • Frozen held-out test discipline (same test for every system).
  • Report accuracy + macro-F1 + per-class F1, not just a single headline number.
  • Label-schema mismatch can be a scientific lever, not just a footnote (old example: AG World ≠ BBC politics).
  • DistilBERT @ max_length=64 felt extremely fast largely because inputs were short; sequence length often dominates model size for latency.

This new project is not another news-topic classifier. It reuses the evaluation mindset and encoder fine-tuning skills, but ships a student product and compares larger / different model classes.

1. Product — useful student note-taking app

One-line pitch: student pastes notes → app classifies the subject → notes are saved into a clean per-subject interface on a website.

Core user flow

  1. User pastes (or types) a note into the web app.
  2. Three classifiers run in parallel: LLM classifier, base BERT, fine-tuned BERT.
  3. An orchestrator LLM reads all three outputs (+ confidences / rationales) and decides the final subject — or chooses Other.
  4. If Other: orchestrator checks the user’s existing custom subjects and assigns one if it fits; otherwise proposes a new custom subject (or a generic Other bucket).
  5. Note is saved under that subject in a clean UI (subject folders / pages, searchable list).
  6. Backend logs the three model votes vs orchestrator decision for research tracking (extra live data; not the primary paper dataset).
Usefulness shift (kept minimal). Same classification research core, but framed as a smart note router: classify → file → organize. Optional later: per-subject actions (summarize, flashcards), but v1 is paste → classify → save cleanly.

UX requirements (v1)

  • Website (desktop + mobile usable).
  • Subject browse: one page/section per subject with saved notes.
  • Show final subject prominently; optionally show the 3 model votes in a “research / details” drawer (do not clutter the primary save flow).
  • User correction control: “Wrong subject?” → move note + log gold label for research.
  • Custom subjects supported via orchestrator Other path.

2. Research question

Primary question: On student-note-like text with a small fixed subject taxonomy, when does a prompted large LLM beat base BERT and fine-tuned BERT-base, and can an orchestrator LLM reliably pick the best vote (or escalate to Other / custom subjects) better than any single system alone?

Secondary angles (optional paper subsections):

  • Proxy-trained BERT (textbooks / exam questions) vs real pasted notes (domain shift).
  • Cost / latency: always-on LLM vs BERT router + LLM only when models disagree or confidence is low.
  • Whether orchestrator over-trusts the LLM vote vs the fine-tuned encoder.

5. Models & size limits

Encoder choice

  • Fine-tuned encoder: bert-base-uncased (Hugging Face; also available as google-bert/bert-base-uncased), AutoModelForSequenceClassification, max_length = 512.
  • Base BERT: same checkpoint family, not fine-tuned on the target train set. Practical definition for a fair head: either (a) linear probe / head-only on a tiny held-out protocol, or (b) zero-shot style via embeddings + label verbalizers — pick one and document it. Do not pretend an untrained 8-way head is meaningful without defining the protocol.
  • Not DistilBERT for the main encoder arm (DistilBERT remains historical context from the prior project).

Token limits (clarified)

ConceptValueMeaning
BERT-base / DistilBERT architecture max 512 tokens Hard model limit; longer text is truncated
Prior UK headlines project setting 64 tokens Project choice for short headlines — not a model limit
This project setting 512 tokens Notes / paragraphs / short textbook chunks

Rough English length at 512 WordPiece tokens: often ~350–400 words. Attention cost grows roughly with length², so 512-token notes will be slower than the old 64-token headline setup even before switching DistilBERT → BERT-base. Expect BERT-base ≈ 1.5–2× slower than DistilBERT at the same sequence length; raising 64 → 512 usually hurts more than that model swap.

LLM arms

  • Classifier LLM: prompted over the 8 subjects (zero-shot + few-shot from train exemplars).
  • Orchestrator LLM: different prompt/role; inputs = note + three votes (+ optional user custom subject list).
  • Prefer a capable API model for research quality; optionally also benchmark a local open model later (repo description mentions GPT-oss as a comparison target).

6. Data for fine-tuning & offline testing

There is no large public corpus of real student notes labeled by subject. Train and freeze-test on educational proxies (exam questions, textbook chunks, short academic explanations). Live app notes are domain-shifted extra data — a feature for the paper, not a problem to hide.

Easiest v1 data glue

  1. Map MMLU tasks → 8 subjects.
  2. Boost STEM with SetFit student-question-categories.
  3. Fill History / Econ / CS / Literature gaps with CogBench + OpenStax/LibreText chunks.
  4. Write a single table: id, text, label, split, source.
  5. Freeze test IDs forever for the paper.

Explicitly out of BERT training

  • Psychology / Philosophy slices (removed subjects) — do not map into the 8.
  • Other / custom subjects — orchestrator only.
  • MMLU noise tasks like miscellaneous, global_facts unless carefully justified.

7. Evaluation & research logging

Offline (paper)

  • Same frozen test for: LLM classifier, base BERT, fine-tuned BERT, orchestrator final.
  • Metrics: accuracy, macro-F1, per-class F1, confusion matrices.
  • Also: latency (p50/p95), token/$ cost for LLM arms.
  • Disagreement analysis: when do the three systems diverge, and does the orchestrator help?
  • Optional coverage-risk / deferral curve if confidence thresholding is added.

Online (extra)

  • Log anonymized: note hash or truncated text (privacy-aware), 3 votes, orchestrator decision, latencies, user correction if any.
  • Do not treat online logs as the primary claimed test set unless users provide explicit gold corrections under a clear protocol.

9. Implementation milestones

  1. Data prep: build unified 8-class corpus; write prepare_data.py; freeze splits; document source mapping.
  2. Fine-tune BERT-base @ 512: train / val early stop; evaluate on frozen test; save checkpoint + metrics JSON.
  3. Base BERT protocol: implement and document the non-finetuned baseline fairly.
  4. LLM classifier prompts: zero-shot + few-shot; score on frozen test.
  5. Orchestrator: prompt that consumes 3 votes + optional custom subjects; score final decisions; measure lift over best single system.
  6. Website v1: paste note → run pipeline → save into subject UI; wrong-subject correction; research event logging.
  7. Paper artifacts: results tables, confusion matrices, latency/cost table, qualitative disagreement examples (including Other/custom cases).
