Shared Cloudflare Tunnel — SocketHR + NoteLMS

Exact instructions for a SocketHR-repo agent: extend the existing Mac Studio Cloudflare Tunnel so api.sockethr.com and api.notelms.com share one cloudflared process. Goal: everyday Mac startup stays at two Express servers + one tunnel command (plus LM Studio GUI), not four commands.

1. Goal & non-goals

Goal

  • Reuse the existing SocketHR Cloudflare Tunnel (same tunnel UUID / credentials).
  • Add a second public hostname: api.notelms.com → a second local Express port.
  • Keep starting cloudflared as one command (same as today).
  • Document the shared-tunnel pattern in sockethr so future agents do not create a second tunnel.

Non-goals

  • Do not create a new Cloudflare Tunnel for NoteLMS.
  • Do not expose LM Studio (127.0.0.1:1234) on the tunnel.
  • Do not merge SocketHR and NoteLMS product APIs into one Express process.
  • Do not attach api.notelms.com (or api.sockethr.com) as a Vercel project domain.
  • Do not implement NoteLMS classify / BERT / orchestrator routes in sockethr.

2. Context from classifier-research / NoteLMS

NoteLMS (repo: classifier-research, site: notelms.com) will mirror SocketHR’s split architecture: Vercel UI + Mac Express API + LM Studio GPT-OSS + Cloudflare Tunnel. The NoteLMS Express server will live in that other repo (or a sibling folder on the Mac), not inside sockethr product routes.

Browser ├─ https://sockethr.com → Vercel (SocketHR UI) ├─ https://notelms.com → Vercel (NoteLMS UI) ├─ https://api.sockethr.com → SAME cloudflared tunnel → 127.0.0.1:3000 (SocketHR Express) └─ https://api.notelms.com → SAME cloudflared tunnel → 127.0.0.1:3002 (NoteLMS Express) └─ either app may call LM Studio on 127.0.0.1:1234 locally
Why sockethr owns this change: the live tunnel config, Path A docs, and deploy/cloudflared/ examples already live in sockethr. NoteLMS only needs a second ingress rule + DNS CNAME pointed at the same tunnel UUID.

3. Port & hostname contract

Lock these unless something already conflicts on the Mac (if so, document the actual ports):

Public hostnameLocal servicePortOwner
api.sockethr.com SocketHR Express 127.0.0.1:3000 sockethr (npm run server)
api.notelms.com NoteLMS Express 127.0.0.1:3002 classifier-research / NoteLMS (separate process)
(not public) LM Studio OpenAI-compatible API 127.0.0.1:1234 LM Studio GUI — localhost only
Port 3001: SocketHR docs sometimes use :3001 for local/dev overrides. Prefer :3002 for NoteLMS to avoid colliding with that convention. If 3002 is taken, pick another free port and use it consistently in tunnel config + NoteLMS server env.

4. What you must change (in sockethr + on the Mac)

4.1 Read current state first

  1. Open the live Mac config (not only the repo example): 
    cat ~/.cloudflared/config.yml
    Confirm it already maps api.sockethr.com → http://127.0.0.1:3000 (or whatever SocketHR actually uses today). Capture the existing tunnel: UUID and credentials-file: path — do not recreate the tunnel.
  2. Confirm the tunnel name / UUID still matches docs, e.g.: 
    cloudflared tunnel list
ls ~/.cloudflared/
  3. Read sockethr docs you will update: docs/STARTUP.md, docs/PUBLIC_TUNNEL.md, docs/PATH_A_SETUP.md, deploy/cloudflared/config.example.yml, deploy/cloudflared/README.md (and any setup script that writes a single-hostname ingress).

4.2 Update live Mac tunnel config

Edit ~/.cloudflared/config.yml so ingress has both hostnames. Keep SocketHR first; keep the catch-all 404 last. 

tunnel: <EXISTING_TUNNEL_UUID>
credentials-file: /Users/<YOUR_USER>/.cloudflared/<EXISTING_TUNNEL_UUID>.json

ingress:
  - hostname: api.sockethr.com
    service: http://127.0.0.1:3000
  - hostname: api.notelms.com
    service: http://127.0.0.1:3002
  - service: http_status:404
Critical: Do not change the tunnel UUID or credentials file. Do not run cloudflared tunnel create … for NoteLMS. Only edit ingress + DNS.

4.3 Restart cloudflared once

Stop the old tunnel process, then start the same way as today, for example:

cloudflared tunnel --config ~/.cloudflared/config.yml run
# or, if you use a named tunnel:
# cloudflared tunnel run sockethr-api

If the dashboard also manages public hostnames for this tunnel, add api.notelms.com → http://127.0.0.1:3002 there as well so dashboard and local config do not fight. Prefer one source of truth (local config.yml for Path A).

4.4 Mirror the example in the sockethr repo

Update deploy/cloudflared/config.example.yml (and any generator script) to show the two-hostname ingress above, with comments explaining:

  • SocketHR owns :3000
  • NoteLMS / classifier-research owns :3002
  • One tunnel process serves both
  • LM Studio stays off the tunnel

5. DNS for api.notelms.com

Same Path A pattern as SocketHR: authoritative DNS for notelms.com is on Vercel; Cloudflare is only the tunnel connector.

  1. Find the existing tunnel UUID from ~/.cloudflared/config.yml (the same one already used by api.sockethr.com).
  2. In the Vercel DNS for the notelms.com project/domain, add:
    TypeNameValue
    CNAME api <SAME_TUNNEL_UUID>.cfargotunnel.com
  3. Do not add api.notelms.com as a domain on the Vercel project (that makes Vercel try to serve it → DEPLOYMENT_NOT_FOUND). DNS record only.
  4. Wait for DNS/SSL. Confirm the CNAME target matches SocketHR’s tunnel UUID exactly (compare to the api.sockethr.com CNAME on sockethr.com).
If you cannot access the notelms.com Vercel DNS from the sockethr agent environment: still complete the tunnel config + sockethr docs, and leave a short docs/NOTELMS_TUNNEL_DNS.md (or section in PUBLIC_TUNNEL.md) with the exact CNAME the human/desktop agent must add. Do not invent a second tunnel as a workaround.

7. Verification

  1. With SocketHR server up and tunnel running: 
    curl -sS https://api.sockethr.com/health
    Expect the existing SocketHR health JSON (must not regress).
  2. With nothing on :3002, https://api.notelms.com/… may 502 — that is OK and proves DNS + ingress reach the Mac. Document that 502 means “tunnel up, NoteLMS server down”.
  3. When a NoteLMS server exists on :3002 with GET /health, verify: 
    curl -sS https://api.notelms.com/health
    Expect JSON like {"ok":true,"service":"notelms-server",…} (exact body owned by NoteLMS).
  4. Confirm LM Studio is still not public: 
    # should NOT be a public hostname in ingress
# curl to any public URL must never hit :1234 directly
  5. Confirm only one cloudflared process is required (no second tunnel UUID in docs or launch scripts).

8. Target everyday startup

StepCommand / actionRequired when
1 LM Studio: load GPT-OSS, start local server :1234 Any AI route
2 SocketHR npm run server → :3000 Using sockethr.com API
3 NoteLMS npm run server → :3002 Using notelms.com API
4 One cloudflared … run Either public API hostname

Acceptable: three terminal commands when both products are in use. Not acceptable: a second long Cloudflare tunnel start command for NoteLMS.

9. Out of scope (do not do)

  • Do not build NoteLMS BERT / classifier / orchestrator endpoints inside sockethr.
  • Do not change SocketHR JWT / Auth.js secrets for NoteLMS (separate apps, separate secrets later).
  • Do not point public/runtime-config.json for sockethr at api.notelms.com.
  • Do not proxy NoteLMS traffic through SocketHR Express “for convenience”.
  • Do not put LM Studio on a public hostname.
  • Do not create cloudflared tunnel create notelms-api (or similar).

10. Done when

  • ~/.cloudflared/config.yml (or documented equivalent) includes both hostnames on the existing tunnel.
  • sockethr repo examples/docs describe the shared tunnel and ports 3000 / 3002.
  • https://api.sockethr.com/health still works with the usual startup.
  • DNS instructions for api.notelms.com → same <uuid>.cfargotunnel.com are written and applied if DNS access is available.
  • STARTUP docs clearly show one cloudflared command for both products.
