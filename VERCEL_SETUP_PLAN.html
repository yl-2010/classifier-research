3. Intended architecture

Model after SocketHR (SOCKETHR_STACK_REFERENCE.html). Short version:

Browser → https://notelms.com (Vercel / Next.js UI + Auth later) │ └─ later: https://api.notelms.com → Cloudflare Tunnel → Mac Studio Express + LM Studio Keep the pretty/public website on Vercel. Keep heavy / private model inference off Vercel (home lab), same split as SocketHR.
  • DO put the Next.js site on the Vercel project that owns notelms.com.
  • DON’T attach an api. subdomain as a Vercel project domain if it should hit the tunnel (SocketHR lesson: that yields DEPLOYMENT_NOT_FOUND).
  • For v1 scaffolding, a placeholder landing page on Vercel is enough; full Auth.js + Mac API can come after project/domain are live.

4. Why this must run on desktop

  • Cloud agent had Vercel MCP in needsAuth state.
  • mcp_auth for Vercel only works in Cursor desktop IDE (interactive OAuth).
  • User logged in on desktop; that session does not transfer into the cloud agent pod.
  • No VERCEL_TOKEN was available in the cloud environment.
  • Therefore this handoff file was written so a desktop agent with working Vercel MCP can finish the work.

6. Exact execution steps

A. Create / link the Vercel project

  1. Using Vercel MCP (preferred) or Vercel dashboard / CLI: create project classifier-research.
  2. Connect Git repository: https://github.com/yl-2010/classifier-research.
  3. Set Framework Preset / Application Preset: Next.js.
  4. If the Next app will live under web/ (recommended, SocketHR-style), set Root Directory to web.
  5. Production branch: main.
  6. Leave build/output as Next defaults once the app exists (next build; no custom output dir needed for standard Next).

B. Attach the domain

  1. Add domain notelms.com to project classifier-research.
  2. Add www.notelms.com and redirect to apex (or vice versa — pick apex as primary).
  3. If DNS is managed by Vercel: use the recommended A/CNAME records Vercel shows.
  4. If DNS is still at the registrar: apply Vercel’s DNS instructions exactly; wait for SSL.
  5. Do not put api.notelms.com on this Vercel project unless the user explicitly wants the API on Vercel (they should not for the SocketHR pattern).

C. Make the repo deployable

  1. Scaffold a minimal Next.js app (see §7).
  2. Commit and push to main (or open a PR if the user prefers — default for this handoff: get production green on main).
  3. Trigger / wait for Vercel production deployment.
  4. Confirm https://notelms.com serves the Next app (even if it’s a placeholder).

D. Report back to the user

  • Vercel project URL (dashboard)
  • Production deployment URL
  • Domain status (valid / pending DNS)
  • What was committed (paths)
  • Any DNS steps the user must still do at the registrar (only if Vercel cannot control DNS)

  • Attach notelms.com
