1. Big picture

SocketHR is not a single monolith on Vercel. It is a split architecture:

  • Cloud Marketing site + dashboard UI + Auth.js Google OAuth on Vercel at https://sockethr.com.
  • Mac Studio Express API (npm run server) on port 3000, talking to LM Studio on 1234, writing jobs/resumes under a local data directory (optionally USB via SOCKETHR_DATA_DIR).
  • Edge Cloudflare Tunnel exposes only the Node API as https://api.sockethr.com. LM Studio stays on localhost and is never public.
Browser (anywhere / LTE) │ ├─ HTTPS https://sockethr.com → Vercel (Next.js Pages Router) │ ├─ Auth.js Google OAuth → /api/auth/[...nextauth] │ ├─ Issues short-lived JWT → GET /api/mac-token │ └─ Fetches /runtime-config.json → apiBase = https://api.sockethr.com │ └─ HTTPS https://api.sockethr.com → Cloudflare Tunnel └─ http://127.0.0.1:3000 → Express (Mac Studio) ├─ verifies JWT (AUTH_SECRET == NEXTAUTH_SECRET) ├─ filesystem storage → SOCKETHR_DATA_DIR / data/jobs └─ OpenAI-compatible HTTP → http://127.0.0.1:1234/v1 └─ LM Studio model → openai/gpt-oss-20b
Design intent: Keep heavy AI + private candidate data on hardware you control. Keep the pretty/public website on Vercel. Bridge them with HTTPS tunnel + shared HMAC secret + short-lived JWTs.

4. Everyday startup (Mac Studio)

Exact production order from docs/STARTUP.md:

  1. LM Studio (GUI) — load openai/gpt-oss-20b, Developer tab → local server ON (port 1234). Confirm: curl -s http://127.0.0.1:1234/v1/models
  2. Express API 
    cd /Users/yanlevin/github/sockethr
npm run server
# or: cd server && node --env-file=.env index.js
    Confirm: curl -s http://127.0.0.1:3000/health → {"ok":true,"service":"sockethr-server",...}
  3. Cloudflare Tunnel 
    cloudflared tunnel --config ~/.cloudflared/config.yml run
# or (dashboard-configured tunnel): cloudflared tunnel run sockethr-api
    Confirm from anywhere: curl -sS https://api.sockethr.com/health

Optional always-on: PM2 or launchd for the Node process; Mac must stay awake. LM Studio must also be running for AI routes.

5. Google sign-in + Mac API JWT bridge

Why two layers?

Vercel owns the browser session (Auth.js / next-auth with Google). The Mac API is a separate origin and cannot see NextAuth cookies. So Next issues a short-lived HS256 JWT that the browser sends as Authorization: Bearer … to the Mac API.

Vercel / NextAuth

  • Provider: Google — src/lib/auth-options.ts
  • Session strategy: JWT (not database sessions)
  • Route: src/pages/api/auth/[...nextauth].ts
  • PKCE note: Google provider uses checks: ["state"] (state-only) because PKCE verifier cookies can fail behind some CDN/proxy setups
  • Secret: NEXTAUTH_SECRET (or AUTH_SECRET)
  • URL: NEXTAUTH_URL=https://sockethr.com
  • Google Cloud Console: OAuth client with redirect https://sockethr.com/api/auth/callback/google (plus localhost callback for local testing)

Mac token minting — GET /api/mac-token

  • File: src/pages/api/mac-token.ts
  • Uses jose SignJWT, alg HS256
  • issuer: sockethr-next
  • audience: sockethr-mac-api
  • TTL: 10 minutes
  • Signed-in: subject = user email; payload includes email, name
  • Guest: cookie sockethr_guest_id (HttpOnly UUID) → subject guest_<uuid>
  • Signing secret: same string as Mac AUTH_SECRET

Mac API verification

  • server/index.js → getAuthFromRequest uses jwtVerify from jose
  • Requires matching issuer/audience and AUTH_SECRET == Vercel NEXTAUTH_SECRET
  • Uploader id = email (or guest subject), lowercased
  • CORS allowlist default: https://sockethr.com, localhost:3000 variants (ALLOWED_ORIGINS)
User clicks Sign in with Google → Auth.js redirects to Google → callback /api/auth/callback/google on Vercel → browser session cookie (NextAuth JWT session) → UI calls GET /api/mac-token (same origin) → UI calls https://api.sockethr.com/api/... with Bearer token → Express jwtVerify → authorize uploader → storage / LM Studio

7. LM Studio / GPT-OSS

Defaults

  • Base URL: http://127.0.0.1:1234/v1
  • Model id: openai/gpt-oss-20b
  • Client: server/lmstudio.js → POST …/chat/completions
  • Env: LM_STUDIO_BASE_URL, LM_STUDIO_MODEL

Security model

  • LM Studio binds to localhost only
  • Only Express is tunnel-exposed
  • Cloud agents / CI without LM Studio: AI routes fail; /health and many unit tests still work

Pattern to copy: one thin OpenAI-compatible wrapper, all product prompts call that wrapper, never call a cloud LLM from the browser for private data. Swap model id via env without code changes. Confirm exact model id with GET /v1/models after loading in LM Studio.

8. Data storage on Mac Studio

  • Default root: <repo>/data/jobs (gitignored)
  • Override: SOCKETHR_DATA_DIR — README mentions USB persistence for resumes/jobs
  • Implementation: server/storage.js (filesystem JSON + resume files, not Postgres)

Layout under data root

SOCKETHR_DATA_DIR/
├── _users/<uploader>.json          # profile, plan, usage, terms
├── _invites/<token>.json           # simulation invite tokens
├── _extension_runs/ …              # LinkedIn import runs
├── _extension_audits/ …

    └── <jobUuid>/
        ├── job.json
        ├── results.json

Resumes without detected applicant email can still be ranked but are not stored under email folders. Path traversal is blocked (UUID job ids, sanitized segments).

Tiers / quotas

server/tiers.js defines Free / Starter / Growth / Scale / Developer. Usage counters live on the user profile JSON. Pre-Stripe trial length is 60 days for paid plans. Developer emails hardcoded for unlimited internal accounts.

10. Cloudflare Tunnel + DNS (Path A — recommended)

SocketHR’s practical Free-plan setup keeps authoritative DNS on Vercel and only uses Cloudflare for the tunnel connector. Full zone / partial CNAME setup notes live in docs/CLOUDFLARE_ZONE_AND_VERCEL.md (partial/CNAME setup needs Business+).

One-time setup

  1. brew install cloudflare/cloudflare/cloudflared
  2. cloudflared tunnel login
  3. cloudflared tunnel create sockethr-api (or bash deploy/cloudflared/setup-path-a.sh)
  4. Write ~/.cloudflared/config.yml from deploy/cloudflared/config.example.yml
  5. At Vercel DNS: CNAME api → <TUNNEL_UUID>.cfargotunnel.com
  6. Do not attach api.sockethr.com as a Vercel project domain (that yields DEPLOYMENT_NOT_FOUND)

Example tunnel config

tunnel: TUNNEL_UUID
credentials-file: /Users/YOUR_USER/.cloudflared/TUNNEL_UUID.json

ingress:
  - hostname: api.sockethr.com
    service: http://127.0.0.1:3000
  - service: http_status:404

Alternatives documented

  • Cloudflare dashboard public hostname (when domain is a Cloudflare zone)
  • ngrok for quick tests
  • Tailscale Funnel
Common failures: 502 = tunnel up but nothing on :3000; 525/SSL = wrong or propagating CNAME; phones calling 127.0.0.1 = runtime-config / env still pointing at loopback (mixed content / wrong host).

11. How the UI finds the API

  1. If NEXT_PUBLIC_SOCKETHR_API_BASE is set at build/runtime → that wins.
  2. Else fetch /runtime-config.json from the website origin (no-store) and read apiBase.
  3. Committed default: { "apiBase": "https://api.sockethr.com" }
  4. Hook: src/lib/useSockethrRuntimeConfig.ts

This lets you change the public API host for phones without rebuilding JS bundles (edit the JSON on the static host / redeploy the public file). Amber banner warns when a non-localhost page resolves an API URL containing localhost.

Extension proxy

Chrome extension prefers same-origin https://sockethr.com/api/extension/*. Next route src/pages/api/extension/[[...path]].ts proxies to the Mac API origin, forwarding Authorization. Direct https://api.sockethr.com also works when tunnel is up.

15. Gotchas agents must respect

  • Do not run npm run build while npm run dev is running — wipe .next if webpack module-not-found appears.
  • HTTPS site cannot call HTTP API (mixed content). Public API must be HTTPS.
  • Never ship apiBase: http://127.0.0.1 for production — phones hit themselves, not the Mac.
  • AUTH_SECRET (Mac) and NEXTAUTH_SECRET (Vercel) must be identical.
  • LM Studio is private; only tunnel the Express port.
  • data/ and server/.env are gitignored — never commit secrets or candidate data.
  • Tunnel credentials live in ~/.cloudflared/<uuid>.json; if lost, delete/recreate tunnel (cannot re-download secret).
  • Waitlist Gmail dedupe: if Cloudflare Email Routing loops mail back to the same Gmail, use a plus-address destination (documented in LOCAL_BACKEND).
  • PDF text is extracted server-side; no vision model required for resumes.

16. Blueprint for a new site in classifier-research

When agents build a new website in this repository, mirror SocketHR’s organization even if product logic differs:

  1. Split packages — web/ (Next.js on Vercel) + server/ (Express on Mac Studio). Root scripts: dev, server, verify:public-api.
  2. Auth — Auth.js Google on Vercel; /api/mac-token (or rename) mints HS256 JWT; Mac verifies with shared secret, issuer, audience.
  3. LLM — LM Studio OpenAI-compatible local server; thin lmstudio.js wrapper; default model pattern like openai/gpt-oss-20b (or whatever you load); never expose port 1234 publicly.
  4. Storage — filesystem under a gitignored data dir; optional external disk via env (*_DATA_DIR). Prefer JSON/files unless you truly need a hosted DB.
  5. Public API hostname — Cloudflare Tunnel + Vercel DNS CNAME api → <uuid>.cfargotunnel.com (Path A). Document a 3-terminal STARTUP.md.
  6. Runtime config — public/runtime-config.json with apiBase, plus optional NEXT_PUBLIC_*_API_BASE override.
  7. Docs agents need — AGENTS.md, docs/STARTUP.md, docs/LOCAL_BACKEND.md, docs/PUBLIC_TUNNEL.md, deploy/cloudflared/ example + setup script, server/.env.example.
  8. Health check contract — GET /health → JSON { ok: true, service: "…" } used by tunnel verify scripts.
  9. CSP / CORS — allow connect to your API host; allowlist browser origins on Express.
  10. Ops — Mac awake, LM Studio + Node + cloudflared after reboot (pm2/launchd), verify from LTE with curl to the public API host.
