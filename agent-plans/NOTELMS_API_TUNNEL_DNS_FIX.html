1. Live diagnosis (why HTTPS fails)

A cloud agent probed production on 2026-07-20. Summary:

CheckResult
https://notelms.com Up (Vercel UI)
https://api.sockethr.com/health Up — Mac + shared tunnel are alive
https://api.notelms.com/health Down — TCP connect fails (no public edge IPs)
notelms.com NS Vercel: ns1.vercel-dns.com / ns2.vercel-dns.com
sockethr.com NS Cloudflare: tia.ns.cloudflare.com / gerardo.ns.cloudflare.com
Vercel DNS api CNAME Exists → 4e5d6603-3e11-4a87-a878-97c0b2a10afb.cfargotunnel.com (confirm this UUID matches the live SocketHR tunnel)
Root cause: Cloudflare Tunnel only publishes reachable edge IPs when the hostname’s DNS record lives in the same Cloudflare account as the tunnel and is proxied (orange cloud). A bare CNAME on Vercel DNS to *.cfargotunnel.com is not enough — clients never get public A/AAAA records, so HTTPS never connects.

That is why resolution looks different today:

HostNameserversWhat dig returns
api.sockethr.com Cloudflare zone Public CF edge IPs (104.21…, 172.67…)
api.notelms.com Vercel DNS CNAME to *.cfargotunnel.com, no public A/AAAA
Older “Path A = Vercel DNS CNAME only” notes in SHARED_CLOUDFLARE_TUNNEL_PLAN.html and docs/PUBLIC_TUNNEL.md are incomplete for the live setup. This file is the authoritative fix plan. After you succeed, update those docs so they match what actually works (Cloudflare-proxied DNS for api).

2. Goal & non-goals

Goal

  • dig +short api.notelms.com A returns public Cloudflare edge IPs (same pattern as SocketHR).
  • curl -fsS https://api.notelms.com/health returns JSON from NoteLMs Express.
  • Reuse the existing SocketHR tunnel UUID / credentials / one cloudflared process.
  • Ingress: api.notelms.com → http://127.0.0.1:3002.

Non-goals

  • Do not create a second Cloudflare Tunnel.
  • Do not expose LM Studio (127.0.0.1:1234) on the tunnel.
  • Do not attach api.notelms.com as a Vercel project domain (causes DEPLOYMENT_NOT_FOUND).
  • Do not merge SocketHR and NoteLMs into one Express process.
  • Do not commit server/.env, server/.auth-secret, or USB user data.

4. Step A — Mac tunnel config

  1. Confirm the live tunnel UUID (must match SocketHR): 
    cat ~/.cloudflared/config.yml
cloudflared tunnel list
ls ~/.cloudflared/
    Compare to the current Vercel CNAME target 4e5d6603-3e11-4a87-a878-97c0b2a10afb. If they differ, the DNS target is wrong — use the UUID from the live config.
  2. Edit ~/.cloudflared/config.yml so ingress has both hostnames. Keep SocketHR first; keep the catch-all 404 last. Do not change tunnel UUID or credentials-file. 
    tunnel: <EXISTING_TUNNEL_UUID>
credentials-file: /Users/<YOU>/.cloudflared/<EXISTING_TUNNEL_UUID>.json

ingress:
  - hostname: api.sockethr.com
    service: http://127.0.0.1:3000
  - hostname: api.notelms.com
    service: http://127.0.0.1:3002
  - service: http_status:404
  3. If Zero Trust dashboard also manages public hostnames for this tunnel, add api.notelms.com → http://127.0.0.1:3002 there too so dashboard and local config do not fight. Prefer one source of truth.
  4. Restart the single tunnel process (do not start a second one): 
    cloudflared tunnel --config ~/.cloudflared/config.yml run
# or: cloudflared tunnel run <existing-tunnel-name>
    Confirm api.sockethr.com/health still works after restart.

5. Step B — DNS (the real fix)

api.notelms.com must resolve like api.sockethr.com: public Cloudflare edge IPs from a proxied record in the tunnel’s Cloudflare account.

Recommended path (matches live SocketHR)

  1. In the same Cloudflare account that owns the SocketHR tunnel, ensure notelms.com can hold a proxied DNS record for api. Practical options:
    • A — Move nameservers: point notelms.com NS to Cloudflare (same pattern as sockethr.com). Keep the Vercel site via Cloudflare DNS records for apex / www (CNAME/flattened to Vercel). Add the tunnel CNAME below.
    • B — Partial / SaaS: only if you already have Cloudflare for SaaS or a partial-zone setup that can place a proxied api.notelms.com into this account without full NS cutover. Harder; do not invent a second tunnel as a workaround.
  2. Add (or correct) this DNS record in Cloudflare:
    TypeNameTargetProxy
    CNAME api <SAME_TUNNEL_UUID>.cfargotunnel.com Proxied (orange cloud)
  3. If Vercel DNS still has an api CNAME after a nameserver move, remove it so it cannot fight Cloudflare. While NS remain on Vercel, a Vercel-only CNAME to *.cfargotunnel.com will not publish public edge IPs — that is the current failure mode.
  4. Do not add api.notelms.com as a domain on the Vercel project. DNS / tunnel hostname only.
  5. Wait for DNS. Success looks like: 
    dig +short api.notelms.com A
# expect Cloudflare anycast IPs (e.g. 104.21… / 172.67…), not only a bare CNAME
# and not an unusable fd10:… ULA
Cloudflare docs: the *.cfargotunnel.com target only proxies traffic for DNS records in the same Cloudflare account as the tunnel. External DNS alone cannot “borrow” the tunnel.

6. Step C — NoteLMs Express on :3002

  1. On the Mac Studio: 
    cd ~/github/classifier-research   # or wherever this repo lives
git pull
npm install --prefix server
npm run server
    Listens on http://127.0.0.1:3002. Defaults: USB data dir, LM Studio GPT-OSS. Auth secret auto-loads from web/.env.local or server/.auth-secret.
  2. Local health (must work before blaming tunnel/DNS): 
    curl -sS http://127.0.0.1:3002/health
  3. Optional for classify/chat: LM Studio GUI with openai/gpt-oss-20b on port 1234. Never put 1234 in tunnel ingress.

7. Verification

# 1) Local API
curl -sS http://127.0.0.1:3002/health

# 2) DNS looks like SocketHR (public CF IPs)
dig +short api.notelms.com A
dig +short api.sockethr.com A

# 3) Public NoteLMs API
curl -fsS https://api.notelms.com/health

# 4) Shared tunnel still serves SocketHR
curl -fsS https://api.sockethr.com/health

# 5) From repo (optional)
npm run verify:public-api
  • 502 on public URL + local health OK = DNS/tunnel OK, Express down — start npm run server.
  • 502 on public URL + nothing on :3002 = same (tunnel up, app down).
  • Connect fail / no public A records = DNS still not Cloudflare-proxied in the tunnel account.
  • 525 / SSL weirdness = wrong hostname registration or propagating DNS.
